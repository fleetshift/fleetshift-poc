# Docker Compose: Design Decisions

This document explains the non-obvious choices in the unified `compose.yaml`
and the Go server changes that support it.

## Unified stack

The compose file runs the full FleetShift stack — backend (Keycloak, Go server)
and frontend (mock plugins, mock API, GUI) — in a single `docker compose up`.
Auth is bootstrapped automatically by an init container (`auth-setup`), so
there are no manual steps after startup.

The UI repo location is configurable via `UI_DIR` in `.env` so the two repos
stay decoupled on disk.

Local Keycloak uses HTTPS as its canonical issuer (`https://keycloak:8443/auth/realms/fleetshift`),
matching external Keycloak behavior. Certificates are generated by `mkcert` on the host
and the CA is automatically trusted by the OS and browsers. See the spec at
`docs/superpowers/specs/2026-06-25-local-keycloak-https-unification-design.md` for details.

## Dual kubeconfig in the kind agent

When the FleetShift server runs inside Docker (via compose), it creates kind
clusters using the host Docker socket (`/var/run/docker.sock`). Kind generates
a kubeconfig with `server: https://127.0.0.1:<nodePort>`, which works from the
host but not from inside the FleetShift container.

The agent fetches two kubeconfigs:

- **External** (`provider.KubeConfig(name, false)`) — `127.0.0.1:<nodePort>`.
  Stored as the cluster's connection info for host-side access.
- **Internal** (`provider.KubeConfig(name, true)`) — `<container-IP>:6443`.
  Used for RBAC bootstrap and ServiceAccount creation from inside the container.

## Module Federation DTS disable

The `@module-federation/dts-plugin` (federated type generation) crashes in
Docker containers — it forks `tsc` processes that fail. Both the GUI and
mock-ui-plugins webpack configs wrap `ModuleFederationPlugin` to pass
`dts: false`:

```typescript
class ModuleFederationPlugin extends BaseMFPlugin {
  constructor(options) {
    super({ ...options, dts: false });
  }
}
```

## JWKS key rotation on restart

`jose`'s `createRemoteJWKSet` caches JWKS keys internally. After
`docker compose down -v` (which destroys the Keycloak volume), Keycloak
generates new signing keys on next startup. The mock API server creates a
fresh `createRemoteJWKSet` on each request to avoid stale keys:

```typescript
function getJWKS() {
  return createRemoteJWKSet(new URL(JWKS_URL));
}
```

This is fine for a dev server. In production you'd use proper JWKS rotation
handling.

## SQLite bind mount

The FleetShift server's SQLite database is bind-mounted to `./data/` on the
host (`./data:/data` in compose). This makes it easy to inspect, back up, or
reset (`rm -rf data/`). The `data/` directory is gitignored.
