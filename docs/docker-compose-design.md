# Docker Compose: Design Decisions

This document explains the non-obvious choices in the unified `docker-compose.yml`
and the Go server changes that support it.

## Unified stack

The compose file runs the full FleetShift stack — backend (Keycloak, Go server)
and frontend (mock plugins, mock API, GUI) — in a single `docker compose up`.
Auth is bootstrapped automatically by an init container (`auth-setup`), so
there are no manual steps after startup.

The UI repo location is configurable via `UI_DIR` in `.env` so the two repos
stay decoupled on disk.

## HTTP Keycloak for browser, HTTPS for kube-apiserver

Keycloak serves on two ports:

| Port | Protocol | Used by |
|------|----------|---------|
| 8180 | HTTP     | Browser (GUI login), mock API server (JWKS verification) |
| 8443 | HTTPS    | kube-apiserver inside kind clusters (`--oidc-issuer-url` requires HTTPS) |

### Why not HTTPS everywhere?

Using HTTPS for browser login requires the self-signed CA to be trusted by the
browser, which is OS-specific and fragile (macOS keychain, Chrome cert store,
Firefox has its own store, etc.). HTTP avoids all of this.

### Why not HTTP everywhere?

Kubernetes API server **requires** `--oidc-issuer-url` to be an HTTPS URL.
There is no flag to disable this — it's hardcoded in the kube-apiserver
validation logic.

### How the split works

The browser's OIDC authority is `http://keycloak:8180/auth/realms/fleetshift`.
The `auth-setup` init container registers this HTTP issuer as the auth method.

When the kind agent creates a cluster, it rewrites the issuer URL before
injecting it into the kubeadm config:

```
http://keycloak:8180/auth/realms/fleetshift
  → https://keycloak:8443/auth/realms/fleetshift
```

This rewrite is controlled by the `OIDC_HTTPS_PORT` env var on the FleetShift
server. When set to `8443`, the agent upgrades HTTP to HTTPS and swaps the port.
When unset, no rewrite happens (original behavior).

The self-signed CA cert is auto-generated by the `keycloak-certs` init container
and mounted into the FleetShift server at `/certs/ca.crt`. The kind agent writes
it to a temp file and mounts it into kind nodes as `--oidc-ca-file`, so the
kube-apiserver trusts it.

### SameSite cookies and the /etc/hosts trick

The browser accesses Keycloak at `http://keycloak:8180`, which resolves to
`127.0.0.1` via `/etc/hosts`. This is critical because:

- The GUI runs at `http://localhost:3000`
- Keycloak sets session cookies on the `keycloak` hostname
- Browsers treat `keycloak → 127.0.0.1` and `localhost → 127.0.0.1` as
  same-site, so SameSite cookies work without the `Secure` flag

If we used `http://localhost:8180` as the issuer instead, the browser would
redirect to `localhost:8180` for login, but then the OIDC issuer (`localhost`)
wouldn't match what containers see (`keycloak`), causing JWT issuer mismatches.

## Dual kubeconfig in the kind agent

When the FleetShift server runs inside Docker (via compose), it creates kind
clusters using the host Docker socket (`/var/run/docker.sock`). Kind generates
a kubeconfig with `server: https://127.0.0.1:<nodePort>`, which works from the
host but not from inside the FleetShift container.

The agent fetches two kubeconfigs:

- **External** (`provider.KubeConfig(name, false)`) — `127.0.0.1:<nodePort>`.
  Stored as the cluster's connection info for host-side access.
- **Internal** (`provider.KubeConfig(name, true)`) — `<container-IP>:6443`.
  Used for RBAC bootstrap and ServiceAccount creation from inside the container.

## CONTAINER_HOST rewriting

When running outside Docker (local dev), kind clusters can't reach `localhost`
OIDC endpoints from inside their Docker network. The `CONTAINER_HOST` env var
(e.g. `host.docker.internal`) rewrites `localhost`/`127.0.0.1` in the issuer
URL to a hostname reachable from containers.

In the compose setup this is set to `keycloak`, but since the issuer is already
`keycloak` the rewrite is a no-op. It's still configured for correctness and
for cases where the issuer might be registered with a localhost URL.

## Module Federation DTS disable

The `@module-federation/dts-plugin` (federated type generation) crashes in
Docker containers — it forks `tsc` processes that fail. Both the GUI and
mock-ui-plugins webpack configs wrap `ModuleFederationPlugin` to pass
`dts: false`:

```typescript
class ModuleFederationPlugin extends BaseMFPlugin {
  constructor(options) {
    super({ ...options, dts: false });
  }
}
```

## JWKS key rotation on restart

`jose`'s `createRemoteJWKSet` caches JWKS keys internally. After
`docker compose down -v` (which destroys the Keycloak volume), Keycloak
generates new signing keys on next startup. The mock API server creates a
fresh `createRemoteJWKSet` on each request to avoid stale keys:

```typescript
function getJWKS() {
  return createRemoteJWKSet(new URL(JWKS_URL));
}
```

This is fine for a dev server. In production you'd use proper JWKS rotation
handling.

## SQLite bind mount

The FleetShift server's SQLite database is bind-mounted to `./data/` on the
host (`./data:/data` in compose). This makes it easy to inspect, back up, or
reset (`rm -rf data/`). The `data/` directory is gitignored.
